** DONE [#A] Choose initial FAISS index type and distance metric
:PROPERTIES:
:Why: The current Redis fallback computes cosine similarity. FAISS supports L2 and maximum inner product search; cosine can be implemented by normalizing vectors and using inner product.
:Change: Use `faiss.IndexIDMap2(faiss.IndexFlatIP(dimension))` for the first implementation. Normalize all passage and query vectors to unit length before adding/searching so inner product equals cosine similarity. Keep IVF/HNSW/PQ out of the first implementation because candidate-scoped passage sets are likely modest and exact search is simpler and deterministic.
:Tests: Add adapter tests showing normalized inner product ranking matches expected cosine ranking and zero vectors are handled without crashes.
:Done when: Index type, metric, and normalization behavior are explicit and test-covered.
:END:

** DONE [#A] Define vector id and metadata sidecar format
:PROPERTIES:
:Why: FAISS ids are numeric, while the domain uses string `passage_id`, `document_id`, source paths, and passage text. The adapter needs a durable mapping.
:Change: Store FAISS vectors in a `.faiss` file and metadata in a SQLite sidecar or JSONL sidecar under `config.app_data_dir / "faiss"`. Prefer SQLite if it avoids loading all metadata at startup and matches existing standard-library persistence. Metadata fields: `vector_id INTEGER PRIMARY KEY`, `passage_id`, `document_id`, `source_path`, `text`, `start_offset`, `end_offset`, `text_hash`, `embedding_model`, `dimension`, and `updated_at`. Generate stable numeric ids from a deterministic hash of `passage_id`, resolving collisions by probing or by keeping an id registry table.
:Tests: Add tests for stable id generation, collision handling if implemented, metadata round trip, and sidecar reopen.
:Done when: The adapter can rebuild domain `EvidencePassage` rows from FAISS search ids without Redis fields.
:END:
